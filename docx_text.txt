Resumen Ejecutivo                     MenuCast es una plataforma SaaS de senalizacion digital disenada para controlar pantallas de negocios en tiempo real. Esta version del documento (v2.0) incorpora datos de una exploracion directa en vivo de la plataforma Yodeck — el lider actual del mercado con mas de 900 partners en 135 paises y calificacion de 4.9/5 — navegando su dashboard autenticado modulo por modulo: Media, Apps, Playlists, Layouts, Schedules y Screens.             El analisis en vivo revela hallazgos criticos que la documentacion publica no muestra: el editor de Layouts es un canvas visual completo tipo Canva (no solo configuracion de zonas), existen 3 tipos de playlists con interactividad para kioscos, un catalogo de 60+ apps/integraciones, un sistema de Takeover Screen para emergencias, y un Web Player que convierte cualquier navegador en pantalla sin hardware.                       ✅  MenuCast YA tiene implementado lo que Yodeck cobra en plan Premium: sincronizacion en tiempo real via WebSockets, cache offline automatico y soporte nativo para Android TV. Arquitectura de Comunicacion Dashboard &lt;-&gt; Pantallas                     Esta es la pregunta central del sistema: como sabe el dashboard el estado de cada pantalla, y como saben las pantallas que contenido mostrar? La respuesta son tres mecanismos combinados:               3.1  Registro y vinculacion (Pairing)             Cuando una pantalla se conecta por primera vez:                      1. El player arranca sin identidad asignada   2. El servidor genera un codigo unico (ej: 'ABC123') -&gt; se muestra en la pantalla   3. El operador ingresa ese codigo en el dashboard   4. El servidor asigna un Screen ID permanente al player   5. El player guarda el Screen ID localmente -&gt; ya tiene identidad             En MenuCast este flujo ya esta implementado en usePairing.ts y PairingScreen.tsx. El pairing_code se genera en el dashboard y el TV lo ingresa para vincularse.               3.2  Heartbeat — el pulso de cada pantalla             Cada player envia una senal de vida al servidor cada 30 segundos (en MenuCast) o cada 60-90 segundos (en Yodeck). Esta senal actualiza: status: 'online', last_seen_at, device_info. Si el servidor no recibe heartbeat por mas de 2x el intervalo, el status pasa a 'offline' automaticamente.               3.3  Sincronizacion de contenido — Pull vs Push                                        Sistema                           Modelo                           Flujo                           Ventaja                           Trade-off                            Yodeck                          Semi-manual Push                          Operador presiona 'Changes pushed'. Latencia: segundos a minutos.                          Simple para el operador                          No es tiempo real. Riesgo de olvidar presionar el boton.                            MenuCast                          Realtime Push automatico                          Cambio en DB -&gt; Supabase Realtime -&gt; player recibe evento -&gt; carga nuevo contenido. Latencia: &lt; 1 segundo.                          Instantaneo, sin accion del usuario                          Conexion WebSocket permanente consume mas recursos en el servidor                   3.4  Cache offline — el safety net                      Pantalla pierde conexion     -&gt; WebSocket se desconecta     -&gt; Player detecta timeout     -&gt; Carga contenido desde AsyncStorage (cache local)     -&gt; Sigue mostrando la ultima version conocida     -&gt; Cuando vuelve internet -&gt; sync automatico en background               3.5  Diagrama de flujo de datos completo                      OPERADOR (Browser / App Movil)       |       |-&gt; Sube imagen/video -&gt; Supabase Storage -&gt; URL publica       |-&gt; Crea Playlist -&gt; tabla playlists       |-&gt; Asigna Playlist a Screen -&gt; screens.current_playlist_id       |       v   SUPABASE (PostgreSQL + Realtime)       |       |-&gt; Evento 'UPDATE' en tabla screens       |       v       |   WebSocket a todos los players suscritos       |   PLAYER (Android TV / Expo)       |-&gt; Recibe evento Realtime       |-&gt; Consulta playlist_items JOIN content_items       |-&gt; Descarga URLs de Supabase Storage       |-&gt; Cachea en AsyncStorage       |-&gt; Reproduce en loop con duracion por item       +-&gt; Heartbeat cada 30s -&gt; UPDATE screens SET status='online'                    4. Ingenieria de Funcionalidades Core                       4.1  Motor de composicion — Layouts multizona (Editor WYSIWYG)             La exploracion en vivo confirma que el editor de Layouts de Yodeck es un canvas visual completo. Para MenuCast, la implementacion debe ser equivalente en UX. Esquema de base de datos:                      -- layouts: plantilla de zonas   CREATE TABLE layouts (     id UUID PRIMARY KEY,     org_id UUID REFERENCES organizations,     name TEXT,     resolution TEXT DEFAULT '16:9-1080p',     background_color TEXT DEFAULT '#000000',     created_at TIMESTAMPTZ DEFAULT NOW()   );    -- layout_zones: cada zona dentro del layout   CREATE TABLE layout_zones (     id UUID PRIMARY KEY,     layout_id UUID REFERENCES layouts ON DELETE CASCADE,     playlist_id UUID REFERENCES playlists,     x DECIMAL NOT NULL,       -- % desde izquierda (0-100)     y DECIMAL NOT NULL,       -- % desde arriba (0-100)     width DECIMAL NOT NULL,   -- % del ancho total (0-100)     height DECIMAL NOT NULL,  -- % del alto total (0-100)     z_index INTEGER DEFAULT 0 -- orden de capas   );    -- layout_elements: texto y shapes decorativos   CREATE TABLE layout_elements (     id UUID PRIMARY KEY,     layout_id UUID REFERENCES layouts ON DELETE CASCADE,     element_type TEXT,  -- 'text' | 'shape' | 'image'     properties JSONB,   -- {content, color, fontSize, shape_type, ...}     x DECIMAL, y DECIMAL, width DECIMAL, height DECIMAL,     z_index INTEGER DEFAULT 1   );               4.2  Motor de programacion — Schedules                      CREATE TABLE schedules (     id UUID PRIMARY KEY,     org_id UUID REFERENCES organizations,     screen_id UUID REFERENCES screens,     playlist_id UUID REFERENCES playlists,     layout_id UUID REFERENCES layouts,     -- opcional     days_of_week INTEGER[],               -- [1,2,3,4,5] = lun-vie     start_time TIME NOT NULL,             -- '07:00:00'     end_time TIME NOT NULL,               -- '11:00:00'     priority INTEGER DEFAULT 0,           -- mayor = gana     active BOOLEAN DEFAULT true   );               4.3  Sistema de Takeover — Anulacion de emergencia             Basado en la exploracion en vivo, el sistema Takeover requiere:                      -- Agregar campos a tabla screens:   ALTER TABLE screens ADD COLUMN     takeover_content_id UUID REFERENCES playlists,     takeover_content_type TEXT,  -- 'playlist' | 'layout'     takeover_expires_at TIMESTAMPTZ,  -- NULL = indefinido     takeover_active BOOLEAN DEFAULT false;    -- Logica del player:   IF screen.takeover_active AND      (screen.takeover_expires_at IS NULL OR       NOW() &lt; screen.takeover_expires_at)   THEN     mostrar takeover_content (ignorar schedules y working_hours)   ELSE     logica normal de schedule/content   END IF;      11.7  Consideraciones tecnicas importantes               Rate limits de APIs externas:         OpenWeatherMap (free): 1000 llamadas/dia. Con 100 pantallas haciendo refresh cada 10min = 14400 llamadas/dia. SOLUCION: cachear la respuesta en Supabase/Redis y hacer la llamada desde el servidor, no desde cada player individualmente.               Offline resilience en apps nativas:         Las apps Tipo B (nativas) deben funcionar sin internet. El reloj funciona siempre. El clima debe cachear la ultima respuesta y mostrarla con indicador de 'actualizacion pendiente'. Las apps Tipo C (API) fallan silenciosamente offline y se pueden ocultar o mostrar placeholder.               Seguridad de credenciales OAuth:         Los tokens OAuth (Google, Instagram, etc.) se almacenan cifrados en la columna oauth_tokens de Supabase, nunca en el player ni en el frontend. El player recibe solo el dato final (los eventos del calendario), no las credenciales.               Preview en el dashboard:         El dashboard debe mostrar un preview de como se vera cada app antes de agregarla a una playlist. Para apps nativas, renderizar un componente React equivalente al de React Native. Para iframe/API, mostrar un screenshot o mockup.               Apps premium vs gratis:         El campo is_premium en app_definitions permite gates por plan. Apps basicas (reloj, QR, WiFi, embed) disponibles en plan free. Apps de integracion (Google, Instagram, Microsoft) solo en plan Pro/Business. Esto crea incentivo de upgrade.                       🗺️  Resumen del camino: Sprint A (framework) + Sprint B (6 apps nativas basicas) = 4 semanas de trabajo. Con eso MenuCast tiene un modulo de Apps funcional con las apps mas demandadas para restaurantes. Los Sprints C y D amplian el catalogo progresivamente sin cambiar la arquitectura base.
